## 一、背景与目的

### 为什么要做这个测试

面试官口中的产品经理 vs. 入职后的真实日常，是两件完全不同的事。

- **面试会说**："你要定义产品方向、驱动业务增长"
- **入职后发现**：你在无尽的需求评审会上和研发 battle，在老板的"灵感"和用户的实际需求之间做翻译，在"这个需求到底做不做"的灵魂拷问中消耗心力

2026年，AI 已经彻底改变了产品经理的工作方式。只会写 PRD、画原型、协调沟通的传统 PM，正在被市场淘汰。而真正稀缺的，是能**用 AI 把想法变成可验证方案**的人。

这个测试的目的：**不是告诉你"能不能做 PM"，而是帮你看到**：

1. 你的能力光谱在 PM 各维度上的分布
2. 你更适配 toC / toB / AI / toM（制造业）/ toG（政府）哪个方向
3. 对于不太适配的方向，如果非要去，需要注意什么
4. 无论结果如何，都能获得有启发性的职业思考

## 二、2026年产品经理核心能力调研汇总

基于对2025-2026年社交媒体讨论、行业报告、企业招聘JD、从业者真实分享的系统梳理，将产品经理核心能力归纳为 **四大域、十二项能力**。

### 2.1 领域一：判断与决策力（"做不做"的能力）

| 能力项 | 说明 | 为什么2026年重要 |
|---|---|---|
| **需求判断与取舍** | 在有限资源下判断什么该做、什么不该做、什么先做 | AI让"能做"的事指数级增加，真正稀缺的是"不做"的判断力 |
| **商业价值判断** | 理解产品如何赚钱、ROI怎么算、商业模式的可持续性 | 投资人不再为"AI概念"买单，PM必须能回答"这功能谁来付费" |
| **风险与边界意识** | 识别法律合规风险（个人信息保护法、AI标识办法等）、技术边界 | 2025年AI标识强制实施后，合规直接决定产品能不能上线 |

### 2.2 领域二：产品设计力（"怎么做"的能力）

| 能力项 | 说明 | 为什么2026年重要 |
|---|---|---|
| **逻辑分析与系统思维** | 拆解复杂问题，设计端到端的业务流程，而非单点功能 | B端产品的核心壁垒就是系统架构的合理性 |
| **用户同理心与需求洞察** | 能从用户行为/反馈中挖掘真实痛点，而非听用户说什么就做什么 | "用户说想要更快的马"但真实需求是"更快到达目的地"——这个能力AI学不会 |
| **产品审美与交互判断** | 对"什么是好产品"有直觉判断，能区分"能用"和"好用" | AI能生成无数方案，但选哪个，靠的是审美 |
| **MVP与快速验证** | 用最小成本验证假设，不追求完美初版 | Vibe Coding让PM能自己做Demo，验证速度决定胜负 |

### 2.3 领域三：技术与数据力（"凭什么"的能力）

| 能力项 | 说明 | 为什么2026年重要 |
|---|---|---|
| **AI技术理解力** | 理解大模型能力边界、Prompt工程、RAG原理、Agent编排基础 | 不会写代码可以，但不懂模型什么时候会"胡说八道"，就没办法做AI产品 |
| **数据思维** | 能定义指标、分析数据、从数据中发现问题和机会 | AI PM还要看准确率、幻觉率、采纳率——不只是DAU/留存 |
| **评测与迭代能力** | 建立评估体系，把"感觉变好了"变成"可被验证地变好了" | AI产品没有评估体系就是黑盒，迭代方向全凭拍脑袋 |

### 2.4 领域四：协作与心力（"走得远"的能力）

| 能力项 | 说明 | 为什么2026年重要 |
|---|---|---|
| **跨角色沟通与推动** | 能在产品、研发、设计、运营、业务方之间高效传递信息并推动共识 | PM没有行政权力但有无限责任，全靠影响力驱动 |
| **抗压与不确定性容忍** | 在信息不完备的情况下做决策，承受来自各方的压力 | AI时代变化更快，没有标准答案，PM必须在混沌中推进 |
| **持续学习与好奇心** | 对新工具、新技术、新行业保持旺盛的学习欲 | 2026年的PM技能体系和2023年已完全不同，不学就掉队 |

## 三、测试方案设计

### 3.1 测试总览

| 维度 | 设计 |
|---|---|
| **题目总量** | 36题（精简版）~ 48题（完整版），建议最终版控制在 **40题左右** |
| **题型** | 全部为单选题，5级李克特量表（非常不符合 / 不太符合 / 一般 / 比较符合 / 非常符合） |
| **预计耗时** | 8~12分钟 |
| **适用终端** | 移动端优先（响应式设计），同时适配桌面端 |
| **技术方案建议** | 单页HTML应用（Vue 3 CDN + 自定义CSS），无需后端，可部署到GitHub Pages/Vercel/Cloudflare Pages |

### 3.2 测试维度与计分矩阵

测试覆盖 **7个能力维度** + **5个方向适配维度**，每道题同时影响多个维度的得分。

#### 7个核心能力维度（每题贡献1~3个维度）

| 编号 | 能力维度 | 缩写 | 对应前文能力 |
|---|---|---|---|
| D1 | 判断取舍力 | JUDGE | 需求判断与取舍、商业价值判断 |
| D2 | 逻辑系统力 | LOGIC | 逻辑分析与系统思维 |
| D3 | 用户洞察力 | EMPATHY | 用户同理心与需求洞察 |
| D4 | 产品审美力 | TASTE | 产品审美与交互判断、MVP验证 |
| D5 | 技术理解力 | TECH | AI技术理解力、数据思维 |
| D6 | 沟通推动力 | COMM | 跨角色沟通与推动 |
| D7 | 心力续航力 | GRIT | 抗压与不确定性容忍、持续学习 |

#### 5个方向适配维度（基于能力组合计算）

| 编号 | 方向 | 简介 | 核心能力组合 |
|---|---|---|---|
| F1 | **toC** | 面向海量个人用户 | EMPATHY + TASTE + 数据思维 |
| F2 | **toB** | 面向企业客户 | LOGIC + TECH + COMM |
| F3 | **AI PM** | AI产品方向 | TECH + JUDGE + 评测能力 |
| F4 | **toM**（中台/内部工具/平台型） | 面向公司内部的效率工具、数据平台、IM系统等；服务"同事即用户"，需求来自多部门 | LOGIC + COMM + TECH + 抽象与平台化思维 |
| F5 | **toG**（政府/公共事业） | 面向政府机构 | COMM + JUDGE + 合规意识 |

### 3.3 题目结构设计

题目按场景分为四大类，穿插排列，避免同类疲劳：

- **A类-情景判断**（约12题）：给一个具体工作场景，问你会怎么做。测判断取舍力、逻辑系统力。
- **B类-偏好与习惯**（约10题）：问你的日常行为偏好。测用户洞察力、产品审美力、心力续航力。
- **C类-技术认知**（约8题）：测你对AI技术的理解和态度。测技术理解力。
- **D类-人际互动**（约10题）：问你在团队中的行为模式。测沟通推动力、协作偏好。

### 3.4 交互设计

> **设计原则**：移动端优先、触控友好、反馈即时、状态可恢复、无阻塞感。所有交互设计基于"用户只需拇指即可完成全部操作"的移动端体验标准。

#### 3.4.1 全局交互规范

| 规范项 | 设计决策 | 理由 |
|---|---|---|
| **选项点击区域** | 最小 48×48px 触控区域，选项卡片整体可点 | 符合 WCAG 2.2 触控目标尺寸要求，拇指操作不误触 |
| **选中反馈** | 点击瞬间：选项卡片边框高亮 + 背景色变化（150ms ease-out）；同时触发轻微 scale(0.97)→scale(1) 回弹 | 即时视觉确认，消除"点了没有？"的不确定感 |
| **防重复点击** | 选中任一选项后，该题所有选项立即 disabled，300ms 内屏蔽所有点击事件 | 避免快速连点导致计分异常 |
| **自动前进** | 选中后延迟 600ms 自动滑入下一题（用户有时间"确认自己的选择"） | 减少操作步骤，保持答题节奏，但不过快导致焦虑 |
| **手动回退** | 顶部导航栏左侧始终可见"上一题"按钮（← 箭头），点击后无动画直接切回，原选择保留高亮 | 用户可以修改答案，避免"手滑点错"的挫折感 |
| **过渡动画** | Vue `<Transition>` 组件，name="slide"，模式为 slide-left（前进）/ slide-right（后退），duration 300ms，easing: cubic-bezier(0.4, 0, 0.2, 1) | 方向感明确，用户始终知道自己在"前进"还是"回退" |
| **键盘支持** | 桌面端：数字键 1-5 快速选择 A-E 选项，左右箭头前进/后退 | 桌面端高级用户效率提升 |
| **无障碍** | 所有选项使用 `<button>` 元素，选中状态通过 aria-pressed 标记；过渡动画尊重 prefers-reduced-motion | 屏幕阅读器友好 |

#### 3.4.3 进度指示器设计

采用"进度条 + 数字"双通道反馈：

- **进度条**：页面顶部 sticky 导航栏下方，3px 高度细线，已完成部分填充主题色（#4F46E5），剩余部分为浅灰（#E5E7EB），带 300ms transition
- **数字**：导航栏右侧显示 `当前题号 / 总题数`（如 `8/40`），字体 13px，颜色 #6B7280
- **自动前进时**：进度条和数字同步更新，视觉上流畅无跳变
- **完成时**：进度条变为绿色（#10B981），数字变为 ✓ 图标，持续 500ms 后进入结果页

#### 3.4.4 状态管理与异常处理

| 场景 | 处理策略 | 实现方式 |
|---|---|---|
| **页面刷新** | 所有已作答题目保存到 localStorage，刷新后恢复答题进度和已选答案 | key: `pm-test-answers`，value: `{1:"B", 2:"C", ...}` |
| **网络断开** | 单页应用不依赖网络，全程可离线使用 | 所有资源内联或 CDN 缓存 |
| **未答完退出** | 再次打开时，弹出提示"你上次答到第X题，是否继续？" | 读取 localStorage，弹窗确认 |
| **结果页回看** | 结果页底部提供"重新测试"按钮，点击清除所有数据回到第1题 | 清除 localStorage，重置 Vue 状态 |
| **分享** | 结果页右上角分享按钮，调用 Web Share API（移动端原生分享面板）或降级为"复制结果图片" | navigator.share() → fallback: html2canvas 截图 → 复制到剪贴板 |

#### 3.4.5 动画规范

| 动画类型 | 持续时间 | 缓动函数 | 说明 |
|---|---|---|---|
| 选项选中反馈 | 150ms | ease-out | 边框 + 背景色过渡 |
| 题目切换（slide-left） | 300ms | cubic-bezier(0.4, 0, 0.2, 1) | 前进动画：旧题左滑消失，新题右滑进入 |
| 题目切换（slide-right） | 300ms | cubic-bezier(0.4, 0, 0.2, 1) | 后退动画：旧题右滑消失，新题左滑进入 |
| 进度条填充 | 300ms | ease-out | 进度条宽度过渡 |
| 雷达图绘制 | 800ms | ease-out | canvas 逐帧绘制 |
| 分数数字跳动 | 1000ms | ease-out | CSS counter 动画，从 0 跳到实际值 |
| 礼花特效 | 2000ms | — | canvas-confetti，结果页加载后触发一次 |

#### 3.4.6 移动端横屏适配

- 横屏时选项卡片改为 2 列布局，避免单列过长
- 题干区域缩减至屏幕高度 30%，确保选项在可视区域内
- 使用 `orientation` media query 检测并切换布局

### 3.5 结果呈现设计

#### 3.5.1 技术选型

基于"单页HTML、CDN引入、无构建工具、移动端优先"的约束，前端工具选型如下：

| 用途 | 工具 | 选型理由 | CDN 引入方式 |
|---|---|---|---|
| **雷达图** | Chart.js v4 | 内置 radar 图表类型，Canvas 渲染性能好，包体积 ~92KB（gzip），移动端友好，学习成本低；支持动画、tooltip、响应式 | `<script src="https://cdn.jsdelivr.net/npm/chart.js@4.4.0/dist/chart.umd.min.js"></script>` |
| **礼花特效** | canvas-confetti | GitHub 12.5k⭐，仅 ~10KB，零依赖，Canvas 高性能粒子渲染，支持自定义颜色/数量/方向，尊重 prefers-reduced-motion | `<script src="https://cdn.jsdelivr.net/npm/canvas-confetti@1.9.4/dist/confetti.browser.min.js"></script>` |
| **分数动画** | 原生 CSS Counter + requestAnimationFrame | 无需额外库，CSS @property 或 JS 缓动函数实现数字递增动画 | 无 |
| **截图分享** | html2canvas（可选） | 如果 Web Share API 不支持，降级为截图分享 | `<script src="https://cdn.jsdelivr.net/npm/html2canvas@1.4.1/dist/html2canvas.min.js"></script>` |

> **为什么不选 ECharts**：功能强大但完整包 ~520KB gzip 后 ~100KB，对于仅需雷达图场景过于沉重。Chart.js 足以满足需求且体积更小。

### 3.6 7维度 × 5方向 权重映射矩阵

> 每个方向的适配度 = Σ(各维度得分 × 该方向对该维度的权重系数) / Σ(权重系数)
> 
> 权重系数基于 2026年各方向PM岗位的 JD 分析和从业者访谈确定，反映该维度在该方向工作中的相对重要性。

#### 3.6.1 权重矩阵（满分 100%）

| 方向 | JUDGE 判断取舍 | LOGIC 逻辑系统 | EMPATHY 用户洞察 | TASTE 产品审美 | TECH 技术理解 | COMM 沟通推动 | GRIT 心力续航 |
|---|---|---|---|---|---|---|---|
| **toC** | 10% | 5% | **30%** | **25%** | 5% | 10% | 15% |
| **toB** | 15% | **25%** | 5% | 5% | 15% | **25%** | 10% |
| **AI PM** | 20% | 10% | 5% | 5% | **35%** | 5% | 20% |
| **toM** | 15% | **30%** | 5% | 5% | 15% | 20% | 10% |
| **toG** | **25%** | 10% | 10% | 5% | 5% | **30%** | 15% |

#### 3.6.2 各方向权重设计解读

**toC（面向个人用户）**——核心是"懂用户 + 好品味"
- EMPATHY(30%) + TASTE(25%) 合计占 55%：C端产品的核心竞争力在于对海量用户需求的洞察和交互审美
- GRIT(15%)：C端竞争白热化，迭代速度快，需要持续学习和抗压能力
- JUDGE(10%) + COMM(10%)：在资源有限的海量需求中做取舍，与运营/设计/研发高频协作
- LOGIC(5%) + TECH(5%)：相对弱需求

**toB（面向企业客户）**——核心是"理清逻辑 + 搞定人"
- LOGIC(25%) + COMM(25%) 合计占 50%：B端产品本质是"把复杂的业务流程系统化"，同时要与多个企业决策者保持高效沟通
- JUDGE(15%) + TECH(15%)：判断客户真实需求 vs. 伪需求，理解企业级技术约束
- GRIT(10%)：B端销售周期长、决策链复杂，需要耐心
- EMPATHY(5%) + TASTE(5%)：B端用户体验要求低于C端

**AI PM（AI产品方向）**——核心是"懂技术 + 敢判断"
- TECH(35%) 权重最高：不懂模型能力边界、Prompt工程、RAG原理就无法做AI产品
- JUDGE(20%) + GRIT(20%)：AI产品变化极快，需要在不确定中持续迭代，判断什么值得投入
- LOGIC(10%)：评测体系设计需要逻辑系统力
- COMM(5%) + EMPATHY(5%) + TASTE(5%)：相对弱需求

**toM（中台/内部平台型）**——核心是"抽象能力 + 多方协调"
- LOGIC(30%) 权重最高：把N个部门的需求抽象成通用平台能力，极其考验系统思维
- COMM(20%)：在多个内部部门之间协调资源、推动共识
- JUDGE(15%) + TECH(15%)：判断优先满足谁的需求，理解平台技术架构
- GRIT(10%)：内部工具难以量化价值，需要长期主义心态
- EMPATHY(5%) + TASTE(5%)：内部用户"被迫使用"，体验要求低于C端

**toG（政府/公共事业）**——核心是"沟通 + 合规判断"
- COMM(30%) 权重最高：toG本质是"关系驱动"，与政府客户保持良好沟通能力是成败关键
- JUDGE(25%)：合规红线判断、风险评估、什么能做什么不能做
- GRIT(15%)：政府采购周期极长（半年到两年），需要极强的耐心和抗压
- EMPATHY(10%)：理解政府用户的使用场景和思维方式
- LOGIC(10%)：政府系统的业务流程设计
- TASTE(5%) + TECH(5%)：相对弱需求

### 5.2 关于 toM（中台/内部平台型PM）的修正说明

toM（Middle Platform / Internal Tools PM）是一个容易被忽略但需求量很大的方向。它的核心特征：

- **用户是内部同事**：研发、运营、客服、销售……而不是外部消费者
- **产品形态**：IM协作工具、数据BI平台、内部CMS、权限管理系统、DevOps平台等
- **核心挑战**：
  - 多部门需求冲突（研发想要X，运营想要Y，老板想要Z，资源有限）
  - 平台化抽象（同一套系统如何服务不同角色的内部用户？）
  - ROI难以量化（"效率提升"怎么算钱？怎么证明价值？）
  - 内部用户的"被迫使用"（他们没得选，但可以不用好）
- **关键能力组合**：逻辑抽象能力（把N个部门的需求抽象成通用平台能力）+ 跨角色沟通 + 技术理解力 + 长期主义心态（内部工具很难"爆"，需要持续打磨）

> 后续开发中，题目的 toM 方向权重需要在计分逻辑中正确映射。
